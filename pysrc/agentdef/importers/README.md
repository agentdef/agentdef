# AgentDef Importers

Importers translate a framework-specific agent definition *into* an
AgentDef directory. They are the reverse of [`adapters/`](../adapters/):

```
Framework-Specific File(s) → Importer → AgentDef Directory
```

This is the manual/scripted counterpart of the
[`framework-import` skill](../../.agentdef/skills/framework-import/SKILL.md)
— same reverse-mapping rules, but deterministic Python instead of an
agent reading the file by hand.

## Available Importers

| Importer | Reads | Command |
| -------- | ----- | ------- |
| [claude](claude/) | `CLAUDE.md` or a Claude Code subagent file (`.claude/agents/<name>.md`, YAML frontmatter) | `agentdef import claude <file> --output <agent-dir>` |
| [copilot](copilot/) | `.github/copilot-instructions.md` (+ optional `.github/instructions/*.instructions.md`) or a `*.agent.md` custom agent | `agentdef import copilot <file> --output <agent-dir>` |
| [m365copilot](m365copilot/) | M365 Copilot declarative agent manifest (`declarativeAgent.json`) | `agentdef import m365copilot <file> --output <agent-dir>` |
| [copilotstudio](copilotstudio/) | Microsoft Copilot Studio agent doc (`<name>.md`, e.g. from github/awesome-copilot-studio-agents) | `agentdef import copilotstudio <file> --output <agent-dir>` |

## Usage

All importers follow the same interface:

```bash
python importers/<framework>/import.py <source-file> --output <agent-directory> [--name <name>]
```

Each run writes a conformant AgentDef directory (`agent.md`, `manifest.yaml`,
`instructions/core.md`, plus whatever skills/workflows/tools/knowledge could
be recovered) **and** an `IMPORT_REPORT.md` listing what mapped cleanly,
what was inferred, and what was dropped because it has no portable AgentDef
equivalent. Always read the report — imports are best-effort, not lossless.

Validate the result the same way you'd validate any AgentDef agent:

```bash
agentdef validate <agent-directory>
```

## Approach

These importers are **deterministic heuristics**, not LLM-driven parsing:
they look for known section headers (Role, Objectives, Style, Priorities,
Avoid, Skills, Workflows, Tools — see `_common.py`'s `_CATEGORY_SYNONYMS`)
in markdown sources, or known JSON fields in structured manifests. Anything
that doesn't match a known pattern is not discarded — it lands in
`instructions/core.md` so no content is silently lost, and the import
report flags it for manual review.

If a source file was generated by one of our own `adapters/` (it carries
the `<!-- Generated from AgentDef. Do not edit manually. -->` marker), the
Claude importer recovers the original structure with much higher fidelity
— see "Round-Trip Check" in the framework-import skill.

The four community-corpus importers (claude, copilot, m365copilot, copilotstudio) have been validated against real-world corpora with 0
failures: 219 agents from github/awesome-copilot (`copilot`), 94 from
github/awesome-copilot-studio-agents (`copilotstudio`), and 187 from
github/awesome-claude-agents + github/awesome-claude-code-subagents
(`claude`, subagent shape) — five real parsing bugs found and fixed along
the way, see CHANGELOG.md.

## Writing Your Own Importer

Each importer is a standalone Python script that:

1. Reads the framework-specific source file(s)
2. Classifies content into AgentDef categories (identity, behavior,
   skills, workflows, tools, knowledge)
3. Writes the result via `_common.AgentDefWriter`, which guarantees the
   output passes `validation/validate.py`

Use `_common.py`'s `generic_markdown_import()` for any new markdown-based
framework (it already knows how to classify common section headers); write
a bespoke parser only for structured formats (JSON/YAML manifests) like
`m365copilot`.
